# AtomS3 (MPU6886) IMU 加速度軸の向き

調査日: 2026-06-17

## 座標系（M5Unified基準）
画面を手前・USB-Cを下にして持った状態を基準とする。

- **X軸**: 右方向（画面の右がプラス）
- **Y軸**: 上方向（画面の上端側がプラス）
- **Z軸**: 画面から手前に飛び出す方向（画面側がプラス）

## 重力の出方
- M5Unified は加速度を **g単位** で返す（9.8ではなく1.0）。
- 画面を上に向けて平置き → 静止時 `accel.z ≈ +1.0`、X/Y ≈ 0。
- 重力はセンサにかかる反力として出るため、Z＋方向（画面の外向き）に +1g が乗る。

## 注意
- 基板リビジョンや M5Unified のバージョンで軸補正が変わる場合がある。
- 最終的には実機で `accel.x/y/z` を出力して符号を確認するのが確実。

## Tab5側の立方体描画への反映 (2026-06-17 追記)
- 画面はY軸下向きのため、device(x,y,z) → model(x,-y,-z) で対応付ける。
- Euler角(roll/pitch)方式をやめ、重力ベクトルを画面下(0,+1,0)へ向ける
  最短回転(Rodrigues)で姿勢を表現。ヨーは原理的に出ず、符号反転バグも消える。
- 遠近の歪み対策: camera distance を 4→8、fov を 500→1000 に。
  立方体半径√3≈1.73に対しcdを十分大きく取ると歪みが減る。
- 検証: 5姿勢で R*grav=(0,1,0)、det=1.0(鏡映なし)を確認済み。

## 透明面/二重面バグの修正 (2026-06-17 追記その2)
- 症状: 青・緑などが二面見える / 面が透明になる。
- 原因: 裏面カリングを「正射影の法線(回転後tv座標)」で判定していたが、
  実際の描画は透視投影。約7%の面でカリング判定と描画がズレ、
  シルエット際の表面が欠けたり裏面が漏れたりしていた。
- 修正: カリングを**透視投影後の画面座標の符号付き面積**で判定。
  画面Y軸は下向きなので、表向きは符号付き面積が負 → `area>=0` を skip。
- 検証: 30万方位で対面リーク0・可視面数は常に1〜3面(破綻なし)を確認。
- 補足: ジャイロは未使用。姿勢はaccelの重力ベクトルのみ、ヨー固定でOK。

## 色化けバグ: TFT_*マクロは使うな (2026-06-17 追記その3)
- 症状: TFT_GREEN が青、TFT_RED が緑、TFT_YELLOW がシアンで描画される。暖色が出ない。
- 原因: M5GFX(LovyanGFX)の `TFT_RED` 等は RGB565値だが **int型**。
  描画関数は uint32_t 引数を **RGB888** として解釈するため、
  565値(例: 緑=0x07E0)が 888(0x0007E0=青)として読まれて化ける。
- 法則: 565値 0x__GG__ → 888解釈で中位バイトが緑、下位が青になる。
  - 赤 0xF800 → 0x00F800 = 緑
  - 緑 0x07E0 → 0x0007E0 = 青
  - 黄 0xFFE0 → 0x00FFE0 = シアン
- 修正: 色は必ず **RGB888 (0xRRGGBB)** リテラルで指定する。
  例: 赤=0xFF0000, 緑=0x00FF00, 青=0x0000FF, 黄=0xFFFF00, シアン=0x00FFFF, マゼンタ=0xFF00FF。
- 注: kEdgeColors(0x800000等)は元々888だったので正しく出ていた。
  TFT_BLACK(=0)は888でも黒なので例外的にそのままで可。
- 未対応: atoms3/src/main.cpp の表示も TFT_* を使用中。同じ化けが起きるはず。
